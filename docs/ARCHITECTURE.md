# 技术计划

## 重整结论

基于真实 KDS 厨房系列报价包，项目不再按“单一后台资料系统”推进，而是按“前台展示 + 后台管理”的一体化业务平台推进。  
当前仓库继续沿用 Vue 3 + Vite + TypeScript + Vben Admin + Supabase 的技术路线，但实施重点调整为：在统一文档、页面设计和数据模型的前提下，持续推进实现、联调、验收和部署准备。

## 当前阶段边界

当前阶段聚焦把平台能力做完整、做稳定、做可交付，实施上遵循以下边界：

- 所有结构性改动必须先与文档口径对齐，再进入 migration、API 和页面实现。
- 继续优先完成核心链路、权限治理、真实数据联调和演示环境收口。
- Excel 自动导入、个人协作扩展等增强能力可以分阶段推进，但不能破坏当前主模型。

当前阶段需要持续固定、不得摇摆的核心结论：

- 产品采用“系列 + 变体”两层。
- 报价采用“批次 + 报价行 + 阶梯价 + 选配项”四层。
- 完整规格存于 `product_spec_items`，常用筛选字段再投影到变体表。
- 原始 Excel 保存为文档对象，不直接等同于报价记录。

## 架构原则

### 前后台同源

当前建议继续使用现有 `apps/web-ele` 作为主应用，在同一工程内规划两套路由和布局：

- 前台布局：偏目录站和商品资料展示。
- 后台布局：偏管理台和数据录入。

这样可以最大化复用当前仓库、权限体系和 API 封装，也符合现有项目约束：

- Supabase Client 统一在 `apps/web-ele/src/lib/supabase.ts` 创建。
- 业务访问统一封装在 `apps/web-ele/src/api/product-info`。

### 后端继续使用 Supabase Cloud

开发阶段仍优先接入 Supabase Cloud，并继续重审旧 migration，避免把基于旧定位的表结构直接当作最终版本执行。

### 前台匿名可见，后台登录管理

虽然前台视觉和信息架构参考 `m.panelook.cn`，但本项目当前采用“前台匿名可见、后台登录管理”的交付方式：

- 匿名用户可读取公开的系列、变体、分类、品牌、公司、文档、报价摘要和动态。
- 匿名用户不可访问后台管理页、上传入口和任何写操作。
- 登录 `user` 角色继续保持只读。
- `profiles.role = 'admin'` 的用户可维护全部业务数据。

## 前端目录约定

Vue 应用在新增页面、文件夹和组件时，默认按业务模块组织，优先适用于 `apps/web-front/src`，其他 Vue 子应用如无特殊约束也应尽量保持一致。

```text
src/
├── views/               # 页面模块，目录名使用 kebab-case
│   └── [module]/        # 业务模块目录
│       ├── index.vue    # 模块主页面
│       ├── detail.vue   # 模块详情页或子页面
│       ├── components/  # 模块私有组件
│       └── [child]/     # 需要继续拆分时的子页面目录
├── components/          # 公共组件
├── composables/         # 组合式函数
├── stores/              # Pinia 状态
├── router/              # 路由配置
├── api/                 # 接口请求
├── utils/               # 工具函数
├── types/               # TypeScript 类型
└── assets/              # 静态资源
```

命名与拆分约束：

- `views` 下的大页面必须使用模块文件夹承载，不再直接平铺 `xxxView.vue`。
- 模块目录使用业务语义命名，不使用 `product-list`、`product-detail` 这类页面结果名；推荐 `product`、`document`、`quote`、`company`。
- 同一业务对象的列表页、详情页、信息页等页面，优先收敛到同一个模块目录内。
- 模块主页面统一使用 `index.vue`，属于该模块的子页面放在同级文件或子目录中。
- 只在当前页面模块内部使用的组件放入该模块的 `components/`；跨页面复用的组件放入 `src/components/`。

## 推荐页面结构

### 前台页面

- `/`
  - 首页，承担平台聚合入口，包含统一搜索、分类入口、热门系列、资料入口、最新报价批次和最新动态。
- `/products`
  - 产品总列表页，按系列展示，支持按分类、品牌、尺寸、芯片、RAM、存储、亮度、POE、报价状态等维度筛选。
- `/categories`
  - 分类导航页，突出目录式入口和分类聚合浏览。
- `/documents`
  - 资料中心，独立展示原始报价工作簿、规格书、图片、图纸、PDF、认证文件和技术资料。
- `/quotes`
  - 报价中心，独立展示报价批次和变体报价行，可按公司、系列、变体、发布日期和状态筛选。
- `/companies`
  - 公司库，展示供应商/品牌方列表，并联动报价总览与平台动态。
- `/updates`
  - 动态信息流，集中展示新品、报价调整、资料更新和内部通知。
- `/products/:seriesId`
  - 系列详情页，采用“系列头部 + 变体切换 + 参数对比 + 完整规格 + 阶梯报价 + 选配说明 + 图片/图纸 + 资料 + 最近动态”的结构。

### 后台页面

- `/admin/dashboard`
  - 统计看板和资料完整度提醒。
- `/admin/products`
  - 系列管理。
- `/admin/products/:seriesId`
  - 系列详情与变体维护。
- `/admin/categories`
  - 分类管理。
- `/admin/brands`
  - 品牌管理。
- `/admin/companies`
  - 公司/供应商管理。
- `/admin/documents`
  - 文档管理。
- `/admin/quotes`
  - 报价批次管理。
- `/admin/quotes/:batchId`
  - 报价批次详情编辑。
- `/admin/updates`
  - 动态管理。
- `/admin/users`
  - 用户和角色管理。

## 数据模型重审

旧方案中的 `products + documents + updates` 只能覆盖一部分需求，不足以表达“一个供应商发布一份报价包，包内包含多个变体报价行、多个阶梯价和多个资料对象”的真实业务。

当前推荐的核心实体包括：

- `categories`
  - 商品分类，支持层级。
- `brands`
  - 品牌或产品线。
- `companies`
  - 供应商、品牌方、报价来源公司。
- `product_series`
  - 系列主表，保存系列级公共信息。
- `product_variants`
  - 变体主表，保存型号、尺寸、芯片、内存、系统等常用筛选字段。
- `product_spec_items`
  - 完整规格真源，保存分组、键值、顺序和展示文案。
- `documents`
  - 扩展后的文档表，保存原始报价工作簿、规格书、图片、图纸、技术文件等。
- `quote_batches`
  - 报价批次主体，表达“这一次报价发布”。
- `quote_lines`
  - 批次内的变体报价行，表达“某型号/变体在本次报价中的配置和备注”。
- `quote_price_tiers`
  - 阶梯价，表达不同数量门槛下的单价。
- `quote_options`
  - 选配项、减价项、固件包、支架等附加信息。
- `updates`
  - 新品、报价更新、资料更新、内部通知。

### 数据设计重点

- `products.spec_json` 只作为兼容字段或缓存字段，不再作为长期主模型。
- 完整规格必须允许按分组原样展示，而不是只做扁平字段。
- 文档必须区分：
  - 原始 workbook
  - 产品图片
  - 尺寸图/结构图
  - 规格资料
  - 认证资料
- 报价主体不再以单一 `unit_price` 为中心，而要以批次和阶梯价为中心。
- 原始 Excel 只是一种来源文档，不直接等同于已结构化的报价记录。
- 文件上传后要能关联到系列、变体、报价批次和动态。

### 动态口径

- 前台动态页、首页最新动态、系列详情最近动态统一以 `updates` 表为唯一数据来源。
- 只有四类事件写入动态：新品发布、报价有效更新、资料可见更新、管理员通知。
- 前台按 `created_at desc` 倒序展示，并过滤掉停用系列、停用变体、不可见资料和非前台有效报价关联的动态。
- 不把草稿保存、备注修改、排序调整、附件上传等后台噪音写入动态。
- 报价动态以 `quote_batches` 和有效报价变更为主来源，不再由报价附件上传直接触发。

## 真实样本抽象

以 `KDS厨房系列新产品 15.6 21.5 32寸报价单 20260428.xlsx` 为例：

- 1 个公司：供应商主体。
- 1 个系列：`KDS 厨房系列`。
- 6 个变体：
  - `KDS1589T`
  - `KDS1585T`
  - `KDS2189T`
  - `KDS2185T`
  - `KDS2489T`
  - `KDS2485T`
- 1 个报价批次：当前报价发布。
- 每个变体 1 条报价行。
- 每条报价行有 4 个阶梯价：`10 / 100 / 500 / 1000`。
- 批次或行级选配信息至少包括：
  - 镀锌板减价
  - EDLA 固件包
  - 摇臂支架
  - 立柱支架

## 实施顺序

### Phase 0：文档统一与实施基线

1. 更新 PRD，明确“报价包”结论和当前阶段边界。
2. 更新架构文档，固定系列/变体、报价批次/报价行/阶梯价模型。
3. 新增独立数据模型文档，沉淀字段、关系、索引和 Excel 映射。
4. 更新路由和 UI 规范，完成页面职责和模块结构说明。

### Phase 1：数据库与 API

1. 按文档产出新的 migration 方案。
2. 梳理兼容旧表与新表的迁移策略。
3. 定义新的前端 API、TypeScript 类型和查询聚合接口。

### Phase 2：后台管理闭环

1. 系列与变体管理。
2. 报价批次、报价行、阶梯价和选配项管理。
3. 文档、图片、图纸和原始报价包管理。
4. 动态发布和可见性口径验证。

### Phase 3：前台展示链路

1. 首页、产品列表、分类导航。
2. 系列详情页。
3. 资料中心、报价中心、公司库和动态页。

## 迁移与兼容要求

- 所有表结构、索引、RLS policy、Storage policy、触发器和函数继续通过 `supabase/migrations` 管理。
- 不在页面中散落直接调用 Supabase Client。
- Storage bucket 继续保持 `product-documents`，除非同时更新 migration、API 层和文档。
- 上线前仍需做一次 Cloud 到本地或自托管环境的迁移演练。

## 当前建议

在完成新模型文档确认前，暂停把当前旧 schema 当成最终版本继续扩展。  
下一步应该优先做的是：

1. 补齐新的业务表设计。
2. 明确前台页面清单和后台页面清单。
3. 统一真实 Excel 到系统对象的映射方式。
4. 文档和设计定稿后，再进入真实 Supabase Cloud 联调和页面开发。
