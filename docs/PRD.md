# 项目 PRD

## 项目背景

项目服务于一家跨境电商 B2B 公司。公司长期经营多种类型商品，但商品规格书、报价单、图片、认证资料、分类信息和供应商信息分散在不同部门、聊天记录和本地文件夹中，造成：

- 销售、产品、采购、运营之间信息不通。
- 报价和资料版本不统一，重复沟通成本高。
- 管理层缺少统一查看、监督和追踪入口。
- 新人很难快速找到可用资料，知识沉淀弱。

## 产品定位

本项目不是电商下单系统，也不是 ERP，更不是普通网盘。  
最终落地形态应是一个带前台展示和后台管理的 B2B 商品信息平台。

一句话定义：
把公司商品、资料、报价和更新动态沉淀为结构化、可检索、可维护、可共享的业务信息资产。

## 目标用户

- 销售：快速查商品资料、报价单、供应商和历史更新。
- 产品/采购：维护分类、参数、文档和报价版本。
- 运营/助理：整理资料、上传文件、发布动态、维护公司信息。
- 管理层：查看商品覆盖情况、资料完整度、报价更新和团队使用情况。

## 使用场景

### 前台展示

前台参考 [m.panelook.cn](https://m.panelook.cn/) 的目录型信息站思路，但不照搬其业务。前台主要承担“浏览、搜索、查阅资料、查看报价和发现更新”的职责，重点页面包括：

- 首页：承担统一入口和按对象搜索入口，搜索默认对象为产品，并分别跳转到产品、资料、报价、公司或动态页面。
- 产品列表：承担商品浏览、产品关键词检索、分类结果和品牌结果，支持分类、品牌、标签、关键参数、资料数量、报价状态等维度筛选商品。
- 分类导航：面向目录式浏览，统一承载产品分类入口和品牌导航入口，帮助用户先按结构再进入商品结果页。
- 商品详情：基础信息、规格参数、相关资料、报价记录、关联公司、最近动态。
- 资料中心 / 报价中心：以独立页面形式查看规格书、图片、报价单和更新记录，并分别承接对应对象的关键词检索。
- 公司库：查看供应商、品牌方和合作公司信息，并联动报价总览和平台动态。
- 动态页：查看新品、报价调整、资料更新和内部通知，并承接动态关键词检索。

前台默认面向公司内部或授权账号开放，展示风格参考公开 B2B 目录站，但数据权限仍由登录和角色控制。

### 后台管理

后台负责全量业务数据维护，重点模块包括：

- 商品管理
- 分类与标签管理
- 公司/供应商管理
- 文档与图片管理
- 报价管理
- 动态与通知管理
- 用户与角色管理

## MVP 范围

- 前台主链路：首页、产品列表、分类导航、资料中心、报价中心、公司库、动态页、商品详情页。
- 商品库：分类、品牌、型号、名称、参数摘要、标签、状态。
- 资料库：规格书、图片、PDF、认证文件、技术资料上传与关联。
- 报价库：按公司和商品管理报价记录、报价单文件、生效时间和状态。
- 公司库：维护报价来源公司、供应商、品牌方基础信息。
- 动态系统：新品发布、报价更新、资料更新、内部通知。
- 搜索系统：按对象搜索，首页搜索默认对象为产品；产品、资料、报价、公司、动态分别跳转各自列表页承接结果。
- 权限系统：`admin` 可写，`user` 只读，匿名不可读写业务表。
- 面向个人协作的前台快捷能力暂不纳入当前 MVP，如收藏、历史、对比、工作台、我的关注、前台上传资料入口、申请报价/联系公司。

## 业务边界

MVP 暂不包含：

- 在线下单、支付、购物车
- ERP 流程、审批流、财务对账
- 复杂 CRM 跟进
- 面向匿名访客的公开商品数据库
- 前台个人协作类能力一次性落地，如收藏、浏览历史、参数对比、个人工作台、我的关注、申请报价/联系公司

## 当前结论

旧方案把项目更偏向“内部资料系统”，方向不算错，但不完整。  
新的最终方向应是：

- 同一套业务数据，服务前台展示和后台管理两套场景。
- 前台体验向 `m.panelook.cn` 靠拢，强调信息导航、参数展示、快速检索和信息聚合入口。
- 后台继续基于现有 Vben 管理能力推进。
- 数据模型不能再把“报价”简单当作普通文档处理，需要作为独立业务对象重点设计。

## 当前阶段

当前仓库已经有商品、文档、动态、用户等原型能力，也有 Supabase migration 草案，但这些内容基于旧定位，需要在真正接入云端前完成一次重审。

当前最重要的不是继续堆页面，而是先统一以下三件事：

1. 前台与后台的信息架构。
2. 商品、报价、资料、公司、动态之间的数据关系。
3. 权限边界和展示范围。
4. 首页热门产品、首页最新动态和独立动态页的首批落地口径。
