# 项目 PRD

## 项目背景

项目服务于一家跨境电商 B2B 公司。公司长期经营多种类型商品，但商品规格书、报价单、图片、认证资料、分类信息和供应商信息分散在不同部门、聊天记录和本地文件夹中，造成：

- 销售、产品、采购、运营之间信息不通。
- 报价和资料版本不统一，重复沟通成本高。
- 管理层缺少统一查看、监督和追踪入口。
- 新人很难快速找到可用资料，知识沉淀弱。

## 产品定位

本项目不是电商下单系统，也不是 ERP，更不是普通网盘。  
最终落地形态应是一个带前台展示、后台管理、资料协同和权限控制能力的 B2B 商品信息业务平台。

一句话定义：  
把公司商品、资料、报价和更新动态沉淀为结构化、可检索、可维护、可共享的业务信息资产。

## 目标用户

- 销售：快速查商品资料、报价单、供应商和历史更新。
- 产品/采购：维护分类、参数、文档和报价版本。
- 运营/助理：整理资料、上传文件、发布动态、维护公司信息。
- 管理层：查看商品覆盖情况、资料完整度、报价更新和团队使用情况。
- 系统管理员：管理用户角色、平台配置、交付环境和权限边界。

## 当前范围说明

- 当前阶段优先交付“前台匿名可见、后台登录管理”的业务链路，先把商品、资料、报价、公司和动态的真实业务链路打通。
- SaaS、多租户、租户隔离、客户自助开通、套餐计费等平台化能力暂不进入当前开发范围。
- 后续如果项目进入稳定运营并开始服务多个客户，再单独规划平台化和 SaaS 化方案。

## 角色与权限模型

- 当前首批权限模型以 `admin` / `user` 为主。
- `admin` 负责维护商品、资料、报价、动态、公司和用户角色。
- `user` 负责登录后查看授权范围内的数据。
- 匿名用户可访问前台公开展示内容，但不可进入后台管理、上传或写操作；RLS 与页面行为必须保持一致。
- 权限控制至少覆盖：
  - 页面访问权限
  - 数据读取权限
  - 新增/编辑/删除权限
  - 启停/发布/归档权限
  - 下载/导出权限
- 后续如果业务复杂度提升，再扩展编辑、审核、报价管理员等细分角色。

## 使用场景

### 前台展示

前台参考 [m.panelook.cn](https://m.panelook.cn/) 的目录型信息站思路，但不照搬其业务。前台主要承担“浏览、搜索、查阅资料、查看报价和发现更新”的职责，重点页面包括：

- 首页：承担统一入口和按对象搜索入口，搜索默认对象为产品，并分别跳转到产品、资料、报价、公司或动态页面。
- 产品列表：承担商品浏览、产品关键词检索、分类结果和品牌结果，支持分类、品牌、标签、关键参数、资料数量、报价状态等维度筛选商品。
- 分类导航：面向目录式浏览，统一承载产品分类入口和品牌导航入口，帮助用户先按结构再进入商品结果页。
- 商品详情：以“系列详情 + 变体切换”为主，集中展示核心参数、完整规格、报价阶梯、选配说明、相关资料、图片/图纸和最近动态。
- 资料中心 / 报价中心：以独立页面形式查看规格书、图片、图纸、报价包和更新记录，并分别承接对应对象的关键词检索。
- 公司库：查看供应商、品牌方和合作公司信息，并联动报价总览和平台动态。
- 动态页：查看新品、报价调整、资料更新和内部通知，并承接动态关键词检索。

前台默认面向匿名访客开放公开数据浏览，展示风格参考公开 B2B 目录站；后台管理、上传与写操作仍由登录和角色控制。

### 后台管理

后台负责全量业务数据维护，重点模块包括：

- 商品系列与变体管理
- 分类与标签管理
- 公司/供应商管理
- 文档、图片与图纸管理
- 报价批次与报价阶梯管理
- 动态与通知管理
- 用户与角色管理
- 基础权限与系统配置管理

## 真实样本校正结论

基于真实文档 `KDS厨房系列新产品 15.6 21.5 32寸报价单 20260428.xlsx`，当前业务对象需要按以下方式重新理解：

- 一份报价单不是“单个附件”，而是一个报价包。
- 一个报价包同时包含：
  - 原始工作簿
  - 报价总表
  - 多个型号规格页
  - 产品图片或尺寸图
  - 阶梯价
  - 选配项或减价说明
- 产品不能只按单表粗放管理，至少要区分“系列”和“变体”两层。
- 报价不能只保留单个 `unit_price`，至少要表达“批次、报价行、阶梯价、选配项”四层结构。
- 规格不能只落在一个松散 JSON 内，必须同时支持：
  - 前台筛选常用字段
  - 规格分组原样展示
  - 后续导入不同供应商模板时的扩展能力

## 平台能力范围

- 前台主链路：首页、产品列表、分类导航、资料中心、报价中心、公司库、动态页、商品详情页。
- 商品库：分类、品牌、系列、变体、参数摘要、完整规格、标签、状态。
- 资料库：原始报价工作簿、规格书、图片、图纸、PDF、认证文件、技术资料上传与关联。
- 报价库：按公司和系列/变体管理报价批次、报价行、阶梯价、选配项、生效时间和状态。
- 公司库：维护报价来源公司、供应商、品牌方基础信息。
- 动态系统：新品发布、报价更新、资料更新、内部通知。
- 搜索系统：按对象搜索，首页搜索默认对象为产品；产品、资料、报价、公司、动态分别跳转各自列表页承接结果。
- 权限系统：支持当前阶段所需的登录、角色区分、前后台访问控制和基础下载授权。
- 状态管理：支持商品、资料、报价的 `active / inactive / archived / draft` 等业务状态控制。
- 运维与交付：支持云端部署、迁移脚本管理、演示环境初始化、角色回归校验和后续私有化迁移准备。
- 面向个人协作的前台快捷能力可分阶段推进，如收藏、历史、对比、工作台、我的关注、前台上传资料入口、申请报价/联系公司；不作为当前首批交付阻塞项。

## 业务边界

当前平台明确不包含：

- 在线下单、支付、购物车
- ERP 流程、审批流、财务对账
- 复杂 CRM 跟进
- 面向匿名访客的公开商品数据库
- 前台个人协作类能力一次性全部落地，如收藏、浏览历史、参数对比、个人工作台、我的关注、申请报价/联系公司
- 首批 Excel 自动导入工作台
- 完整 ERP、CRM、工单、财务和订单履约一体化闭环
- SaaS 多租户、客户自助开通、套餐计费和平台商业化能力
- AI 解析、AI 搜索、AI 问答、AI 质检等智能化能力

## 后续预留方向

- 如果项目后续进入稳定运营并开始复制到多个客户，再规划 SaaS、多租户、租户隔离、商业化计费等平台能力。
- 如果资料规模和整理成本显著上升，再评估 AI 解析、AI 搜索、AI 质检、AI 问答等智能化能力。
- 如果多人协作复杂度进一步提升，再规划审计日志、通知中心、版本回溯和轻量审核流。

## 当前阶段

当前仓库已经有商品、文档、动态、用户等基础能力，也有 Supabase migration 草案，但其中部分内容基于旧定位，需要在持续实现过程中继续收敛和升级。

当前阶段不是停留在“文档与设计定稿”，而是在统一文档、设计和数据模型的基础上，持续推进前后台实现、联调、验收和部署准备。  
在本阶段，必须持续确保以下内容同步收敛：

1. 前台与后台的信息架构。
2. 商品、报价、资料、公司、动态之间的数据关系。
3. 系列、变体、报价批次、报价行、阶梯价、选配项、原始报价包的定义边界。
4. 首页热门系列、首页最新报价批次、商品详情页和报价中心的首批展示口径。
5. 文档、架构、页面设计之间的命名和结构一致性。
6. 前后台真实使用链路、权限口径和交付边界的统一定义。

## 当前结论

旧方案把项目更偏向“内部资料系统”，方向不算错，但不完整。  
新的最终方向应是：

- 同一套业务数据，服务前台展示和后台管理两套场景。
- 前台体验向 `m.panelook.cn` 靠拢，强调信息导航、参数展示、快速检索和信息聚合入口。
- 后台继续基于现有 Vben 管理能力推进，并优先补齐真实业务维护所需的效率和稳定性。
- 数据模型不能再把“报价”简单当作普通文档处理，需要作为独立业务对象重点设计。
- 真实报价包应驱动新的主模型，而不是让页面继续围绕旧原型字段扩展。
- 当前交付目标不是做 SaaS 平台，而是先完成一个可部署、可演示、可交付、可支撑真实业务使用的业务系统。
- SaaS、多租户、AI 和平台商业化能力统一放到后续阶段再评估，不作为当前开发阻塞项。

## 本阶段完成标准

- `docs/PRD.md`、`docs/ARCHITECTURE.md`、`docs/DATA_MODEL.md`、`docs/ROUTES.md`、`design/frontend-ui/README.md`、`design/frontend-ui/UI_SPEC.md` 对同一套对象命名保持一致。
- 任一开发者只看文档，就能明确：
  - 该建哪些表
  - 页面展示哪些字段
  - 列表和详情按什么对象组织
  - 原始 Excel 中的真实字段如何落库
- 文档、实现和验收并行推进，但所有新增开发都必须以已统一的文档口径为准。
