# 后台联动与流程优化任务清单

本文档用于收口当前“后台管理流程、前台联动、录入体验优化”的专项任务，作为接下来产品、设计、开发联动的执行清单。

当前明确决策：

- 前台改为“匿名可见”。
- 后台录入流程必须从“分散多页面跳转维护”改为“围绕一次商品建档或一次报价发布的工作流”。
- 后续实现优先围绕“系列中心、批次中心、低跳转、可批量、可 AI 辅助”推进。

状态约定：

- `[x]` 已完成
- `[ ]` 待开发
- `[-]` 暂不进入当前批次

## Priority 1：前台改为匿名可见

目标：统一文档、权限和页面行为，明确前台为公开浏览入口，但后台写操作仍严格受 RLS 和角色控制。

- `[ ]` 同步更新文档口径：将 `docs/ARCHITECTURE.md`、`docs/PRD.md`、`docs/README.md` 中“前台不是匿名公开站”的描述改为“前台匿名可见，后台登录管理”。
- `[ ]` 复核并确认 `supabase/migrations/005_frontend_anonymous_read_access.sql` 作为正式策略，而不是临时策略。
- `[ ]` 梳理匿名可见范围：产品、分类、品牌、公司、资料、报价摘要、动态可见；后台管理页、上传、写操作不可见。
- `[ ]` 前台增加匿名访问视角验收：未登录状态下可直接打开首页、产品列表、详情页、资料中心、报价中心、公司库、动态页。
- `[ ]` 后台和 Supabase 权限回归：匿名仅可读公开数据；`user` 仅读；`admin` 可写。

验收标准：

- 未登录用户可直接浏览前台核心页面。
- 匿名访问不会看到后台入口、编辑按钮和写操作能力。
- RLS 与 Storage policy 口径和文档完全一致。

## Priority 2：后台流程重构，降低录入成本

目标：把当前“先建分类/品牌/公司，再建产品，再传资料，再建报价”的割裂流程，改造成围绕业务动作的一站式流程。

### 2.1 流程重构方向

- `[ ]` 新增“商品建档工作台”方案设计。
- `[ ]` 新增“报价发布工作台”方案设计。
- `[ ]` 明确后台默认入口不再是孤立的表格页，而是“新建商品 / 发布报价 / 上传资料”三个主动作入口。

建议方案：

1. 商品建档工作台
   - 第一步：输入型号或系列名。
   - 第二步：自动推荐或快速创建分类、品牌、公司。
   - 第三步：补产品基础信息与关键规格。
   - 第四步：上传主图、规格书、技术资料。
   - 第五步：确认前台展示效果并发布。

2. 报价发布工作台
   - 第一步：选择公司和系列。
   - 第二步：选择或批量勾选多个变体。
   - 第三步：录入批次信息、阶梯价、选配项。
   - 第四步：上传报价附件并自动关联批次。
   - 第五步：预览前台报价展示和动态，再发布。

### 2.2 快速创建能力

- `[ ]` 在产品建档弹窗内支持“就地新增分类 / 品牌 / 公司”，不强制跳出到独立管理页。
- `[ ]` 在文件上传页支持根据当前产品或当前批次自动回填关联项。
- `[ ]` 在报价工作台支持从系列自动带出变体，不再逐条手选。
- `[ ]` 增加“保存为草稿，稍后继续”的中间状态。

### 2.3 批量能力

- `[ ]` 设计“系列级批量建变体”功能，支持按尺寸、内存、存储等组合快速生成变体草稿。
- `[ ]` 设计“报价批次批量录入”功能，支持一张表内维护多个变体的阶梯价。
- `[ ]` 设计“资料批量上传”功能，上传后按文件名规则自动猜测型号、资料类型和归属批次。

### 2.4 AI 辅助方向

- `[ ]` 增加“AI 识别报价单 / 规格书 / 命名文件”的方案设计。
- `[ ]` 增加“AI 自动推荐分类、品牌、公司、资料类型”的方案设计。
- `[ ]` 增加“AI 生成结构化草稿，人工确认后入库”的方案设计。

建议 AI 落地方向：

1. 文件上传后先做结构化抽取
   - 从规格书抽取型号、尺寸、分辨率、芯片、RAM、存储、系统。
   - 从报价单抽取批次标题、币种、阶梯价、选配项。

2. 生成待确认草稿
   - 自动匹配已有分类、品牌、公司、系列、变体。
   - 匹配不到时生成“建议新建”项。

3. 人工二次确认
   - 管理员只需要勾选“采用 / 修改 / 忽略”，而不是从零手填。

验收标准：

- 单个新产品从录入到前台可见，页面跳转次数明显下降。
- 单个报价批次发布不再依赖多个后台页面来回切换。
- AI 只做“识别和建议”，最终入库仍由人工确认。

## Priority 3：改成“系列中心 + 批次中心”的工作方式

目标：让后台结构和前台展示结构一致，避免继续围绕单变体和平铺表格扩展。

- `[ ]` 规划 `/admin/products/:seriesId` 系列详情页，统一维护系列基础信息、变体、规格、图片、资料和关联公司。
- `[ ]` 规划 `/admin/quotes/:batchId` 报价批次详情页，统一维护批次头信息、报价行、阶梯价、选配项和附件。
- `[ ]` 前台产品详情改为系列详情，不再以单变体页面作为长期主形态。
- `[ ]` 明确“系列”和“变体”的前后台展示规则：前台详情主对象是系列，变体作为切换器；后台录入主对象也是系列。
- `[ ]` 补齐 `quote_options` 的后台维护入口，不能只读不写。

建议实施方案：

1. 后台产品列表页保留“系列列表”。
2. 点进系列后，在一个页面内完成：
   - 系列信息维护
   - 变体列表维护
   - 规格维护
   - 图片和资料维护
   - 前台预览
3. 后台报价列表页保留“批次列表”。
4. 点进批次后，在一个页面内完成：
   - 批次信息
   - 多变体报价行
   - 阶梯价
   - 选配项
   - 附件
   - 动态预览

验收标准：

- 后台产品和报价的主对象与数据模型保持一致。
- 前台详情和后台详情围绕同一业务对象，不再各自维护一套理解。

## Priority 4：统一查询层和动态治理

目标：减少前后台两套口径继续漂移，并让动态只展示对前台真正有意义的信息。

### 4.1 查询层统一

- `[ ]` 梳理 `apps/web-front/src/api/product-info.ts` 与 `apps/web-ele/src/api/product-info/*` 的重复逻辑。
- `[ ]` 设计统一的聚合查询层或共享查询模块，统一产品、详情、报价、资料、动态的口径。
- `[ ]` 统一“active / archived / expired / draft”的前后台解释，避免一个页面显示、另一个页面隐藏。

建议方案：

1. 后台 API 负责写操作和管理态列表。
2. 前台 API 负责公开态聚合查询，但底层字段映射和状态规则抽到共享层。
3. 系列详情、报价批次详情、动态卡片都走统一聚合模型。

### 4.2 动态治理

- `[ ]` 重写动态生成规则，只保留“新品发布 / 有效报价更新 / 可见资料更新 / 管理员通知”四类。
- `[ ]` 取消“仅因报价附件上传就生成报价动态”的旧逻辑。
- `[ ]` 把“发布报价批次”作为报价动态的主要来源。
- `[ ]` 增加动态可见性规则：停用系列、停用变体、不可见资料、无效报价不进入前台动态流。
- `[ ]` 后台增加“是否生成前台动态”的确认或预览能力。

验收标准：

- 前台动态不再出现重复、噪音、误导信息。
- 动态内容能真实反映“产品发布、资料上新、报价更新、平台通知”。

## 推荐实施顺序

1. 先统一 Priority 1 文档和权限口径。
2. 再做 Priority 2 的工作流原型和页面方案。
3. 同步推进 Priority 3 的系列详情页、报价批次详情页。
4. 最后用 Priority 4 收敛查询层和动态口径。
