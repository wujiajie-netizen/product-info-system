# 后台联动与流程优化任务清单

本文档只保留当前仍需开发或继续完善的“后台管理流程、前台联动、录入体验优化”任务。已经完成并验收通过的事项不再在本文档中保留。

当前明确决策：

- 前台改为“匿名可见”。
- 后台录入流程从“分散多页面跳转维护”改为“围绕一次商品建档或一次报价发布的工作流”。
- 后续实现优先围绕“系列中心、批次中心、低跳转、可批量、可 AI 辅助”推进。
- `/admin/quotes` 继续命名为“报价管理”，主形态为“列表进入报价批次详情页”。
- 报价选配项不再作为独立菜单入口，长期维护合并到报价批次详情页。

状态约定：

- `[ ]` 待开发
- `[-]` 暂不进入当前批次

## Priority 1：前台改为匿名可见

目标：统一文档、权限和页面行为，明确前台为公开浏览入口，但后台写操作仍严格受 RLS 和角色控制。

- `[ ]` 同步更新文档口径：将 `docs/ARCHITECTURE.md`、`docs/PRD.md`、`docs/README.md` 中“前台不是匿名公开站”的描述改为“前台匿名可见，后台登录管理”。
- `[ ]` 复核并确认 `supabase/migrations/005_frontend_anonymous_read_access.sql` 作为正式策略，而不是临时策略。
- `[ ]` 梳理匿名可见范围：产品、分类、品牌、公司、资料、报价摘要、动态可见；后台管理页、上传、写操作不可见。
- `[ ]` 前台增加匿名访问视角验收：未登录状态下可直接打开首页、产品列表、详情页、资料中心、报价中心、公司库、动态页。
- `[ ]` 后台和 Supabase 权限回归：匿名仅可读公开数据；`user` 仅读；`admin` 可写。

验收标准：

- 未登录用户可直接浏览前台核心页面。
- 匿名访问不会看到后台入口、编辑按钮和写操作能力。
- RLS 与 Storage policy 口径和文档完全一致。

## Priority 2：Excel 商品快速入库工作台后续增强

目标：在现有 Excel 商品快速入库工作台基础上，继续提升稳定性、可复用性、可追溯性和业务可控性。已完成的原始 Excel 报价附件、Excel 嵌入图片上传、快速创建分类 / 品牌 / 公司不再作为待办保留。

边界说明：

- Priority 2 负责“导入、创建、快速补齐”。
- Priority 3 负责“创建后的标准维护”。
- Priority 4 负责“统一数据口径和前台展示规则”。
- Priority 2 可以在导入时创建商品、报价、阶梯价、资料和选配项草稿，但不应在工作台里实现完整的长期详情维护页面。
- Priority 2 如需触发新品发布、报价更新、资料更新等前台动态，只能提供动作标记或预览入口，最终是否生成动态、生成哪类动态，必须遵守 Priority 4 的动态治理规则。

### 2.1 导入预检报告继续增强

目标：在真正写入数据库前，给管理员一个清晰的导入风险与结果预览，避免误导入、重复导入和部分数据缺失。

待开发：

- `[ ]` 进一步校准预检结果与实际入库结果数量一致性。
- `[ ]` 增强缺规格、缺图片、缺报价等异常类型的分组展示。
- `[ ]` 增强错误报告内容，支持导出更完整的行级失败原因和处理建议。

验收标准：

- 管理员在点击“确认入库”前，可以明确知道本次会新增、更新、跳过哪些数据。
- 异常行不会阻塞正常行入库。
- 预检结果与实际入库结果数量一致。

### 2.2 重复型号处理继续增强

目标：解决“商品已存在，但报价 Excel 又重新导入”的真实业务场景，避免重复创建商品。

待开发：

- `[ ]` 更新已有商品时，支持逐字段人工确认。
- `[ ]` 重复型号检测展示更完整的已有商品信息：型号、名称、系列、分类、品牌、公司、更新时间。
- `[ ]` 对“作为新商品创建”的同型号场景增加二次确认，避免误创建重复产品。

字段覆盖策略：

- 保留已有值，Excel 只补空字段。
- 使用 Excel 覆盖已有值。
- 逐字段人工确认。

验收标准：

- 重复型号不会默认重复创建商品。
- 同一型号新报价可以直接挂到已有商品。
- 更新已有商品时不会误删已有资料、报价和关联关系。

### 2.3 报价阶梯可编辑

目标：管理员可以在工作台内修正价格，不必回到 Excel 修改后重新上传。

待开发：

- `[ ]` 将当前报价阶梯展示改成可编辑表格。
- `[ ]` 支持编辑 10 / 100 / 500 / 1000 等阶梯价格。
- `[ ]` 支持新增自定义阶梯数量，例如 50、200、2000。
- `[ ]` 支持删除某个阶梯价格。
- `[ ]` 支持批量设置币种。
- `[ ]` 支持批量设置 MOQ。
- `[ ]` 支持空价格自动忽略，不生成对应阶梯价。
- `[ ]` 支持价格格式校验，只允许数字和有效小数。
- `[ ]` 支持在报价确认区按型号展示报价明细。

验收标准：

- 报价解析错误时，可以在工作台中直接修正。
- 编辑后的价格以管理员确认后的数据为准。
- 入库后的报价阶梯与页面确认表格一致。

### 2.7 品牌、公司、分类自动匹配

目标：减少人工选择默认值，提高 Excel 导入效率。

待开发：

- `[ ]` 从 Excel 文件名匹配系列、分类、品牌、公司。
- `[ ]` 从报价 Sheet 顶部公司名匹配已有公司。
- `[ ]` 从详情 Sheet、配置文本、标题中匹配品牌和分类关键词。
- `[ ]` 支持品牌别名匹配。
- `[ ]` 支持公司 slug、英文名、中文名模糊匹配。
- `[ ]` 匹配结果显示置信提示：自动匹配、疑似匹配、未匹配。
- `[ ]` 疑似匹配必须由管理员确认后才可入库。
- `[ ]` 支持批量接受自动匹配结果。

验收标准：

- 常见供应商 Excel 上传后，可以自动带出公司。
- 同类产品 Excel 上传后，可以自动推荐分类。
- 自动匹配不会静默创建新数据。

### 2.8 Excel 模板映射配置

目标：让工作台不只适配 KDS 这一类报价单，而是支持不同供应商、不同列名、不同 Sheet 结构的结构化 Excel。

待开发：

- `[ ]` 支持从当前 Excel 生成模板。
- `[ ]` 支持配置更完整的报价 Sheet 名称或匹配规则。
- `[ ]` 支持配置表头所在行。
- `[ ]` 支持配置型号列、尺寸列、标准配置列、备注列。
- `[ ]` 支持配置价格阶梯列：列名、起订量、币种。
- `[ ]` 支持配置型号详情 Sheet 识别规则。
- `[ ]` 支持配置详情 Sheet 字段映射：Excel 标签 → 系统字段。
- `[ ]` 支持模板复制。
- `[ ]` 支持模板试解析：上传样例 Excel 后查看解析结果。

验收标准：

- 新供应商 Excel 只要结构化，就可以通过模板配置解析。
- 模板配置后，同类 Excel 后续导入无需重新映射。
- 模板解析结果必须进入人工确认表格，不允许直接静默入库。

### 2.10 规格字段结构化增强

目标：将 Excel 详情 Sheet 中的规格字段从 `spec_json` 进一步沉淀到可筛选、可展示、可维护的规格项。

验收标准：

### 2.12 导入历史与失败日志继续增强

## Priority 3：改成“系列中心 + 批次中心”的工作方式

目标：让后台结构和前台展示结构一致，避免继续围绕单变体和平铺表格扩展。

待开发：

- `[ ]` 完善 `/admin/products/:seriesId` 系列详情页，统一维护系列基础信息、变体、规格、图片、资料和关联公司。
- `[ ]` 前台产品详情改为系列详情，不再以单变体页面作为长期主形态。
- `[ ]` 明确“系列”和“变体”的前后台展示规则：前台详情主对象是系列，变体作为切换器；后台录入主对象也是系列。

建议实施方案：

1. 后台产品列表页保留“系列列表”。
2. 点进系列后，在一个页面内完成：系列信息维护、变体列表维护、规格维护、图片和资料维护、前台预览。
3. 后台报价列表页保留“批次列表”。
4. 点进批次后，在一个页面内完成：批次信息、多变体报价行、阶梯价、选配项、附件、动态预览。

验收标准：

- 后台产品和报价的主对象与数据模型保持一致。
- 前台详情和后台详情围绕同一业务对象，不再各自维护一套理解。

## Priority 4：统一查询层和动态治理

目标：减少前后台两套口径继续漂移，并让动态只展示对前台真正有意义的信息。

### 4.1 查询层统一

- `[ ]` 梳理 `apps/web-front/src/api/product-info.ts` 与 `apps/web-ele/src/api/product-info/*` 的重复逻辑。
- `[ ]` 设计统一的聚合查询层或共享查询模块，统一产品、详情、报价、资料、动态的口径。
- `[ ]` 统一“active / archived / expired / draft”的前后台解释，避免一个页面显示、另一个页面隐藏。

建议方案：

1. 后台 API 负责写操作和管理态列表。
2. 前台 API 负责公开态聚合查询，但底层字段映射和状态规则抽到共享层。
3. 系列详情、报价批次详情、动态卡片都走统一聚合模型。

### 4.2 动态治理继续增强

- `[ ]` 重写动态生成规则，只保留“新品发布 / 有效报价更新 / 可见资料更新 / 管理员通知”四类。
- `[ ]` 取消“仅因报价附件上传就生成报价动态”的旧逻辑。
- `[ ]` 把“发布报价批次”作为报价动态的主要来源。
- `[ ]` 后台增加“是否生成前台动态”的确认或预览能力。
- `[ ]` 将动态治理规则接入更多前台查询和后台发布入口。

验收标准：

- 前台动态不再出现重复、噪音、误导信息。
- 动态内容能真实反映“产品发布、资料上新、报价更新、平台通知”。

## 推荐实施顺序
