# 后台联动与流程优化任务清单

本文档用于收口当前“后台管理流程、前台联动、录入体验优化”的专项任务，作为接下来产品、设计、开发联动的执行清单。

当前明确决策：

- 前台改为“匿名可见”。
- 后台录入流程必须从“分散多页面跳转维护”改为“围绕一次商品建档或一次报价发布的工作流”。
- 后续实现优先围绕“系列中心、批次中心、低跳转、可批量、可 AI 辅助”推进。

状态约定：

- `[ ]` 待开发
- `[-]` 暂不进入当前批次

## Priority 1：前台改为匿名可见

目标：统一文档、权限和页面行为，明确前台为公开浏览入口，但后台写操作仍严格受 RLS 和角色控制。

- `[ ]` 同步更新文档口径：将 `docs/ARCHITECTURE.md`、`docs/PRD.md`、`docs/README.md` 中“前台不是匿名公开站”的描述改为“前台匿名可见，后台登录管理”。
- `[ ]` 复核并确认 `supabase/migrations/005_frontend_anonymous_read_access.sql` 作为正式策略，而不是临时策略。
- `[ ]` 梳理匿名可见范围：产品、分类、品牌、公司、资料、报价摘要、动态可见；后台管理页、上传、写操作不可见。
- `[ ]` 前台增加匿名访问视角验收：未登录状态下可直接打开首页、产品列表、详情页、资料中心、报价中心、公司库、动态页。
- `[ ]` 后台和 Supabase 权限回归：匿名仅可读公开数据；`user` 仅读；`admin` 可写。

验收标准：

- 未登录用户可直接浏览前台核心页面。
- 匿名访问不会看到后台入口、编辑按钮和写操作能力。
- RLS 与 Storage policy 口径和文档完全一致。

## Priority 2：Excel 商品快速入库工作台后续增强

目标：在现有 Excel 商品快速入库工作台基础上，继续提升稳定性、可复用性、可追溯性和业务可控性。当前阶段不接入 AI，优先把结构化 Excel 导入做稳，支持更多供应商报价模板、重复型号处理、报价编辑、附件留档、图片提取和导入历史。

当前策略：

- 继续保留现有“一个页面完成上传、解析、确认、入库”的工作台形态。
- 不推翻现有产品管理、文件资料、报价管理页面。
- 后续功能优先复用现有产品、报价、资料 API。
- Excel 结构化解析不依赖 AI；AI 仅作为后续非结构化 PDF、图片报价单、图片规格书的增强方向。

### 2.0 与 Priority 3 / Priority 4 的边界说明

Priority 2、Priority 3、Priority 4 不互相替代，三者职责边界如下：

- Priority 2 负责“导入、创建、快速补齐”。它是业务录入入口，重点解决 Excel 导入、手动快速建档、资料补齐、报价录入、预检确认和批量入库。
- Priority 3 负责“创建后的标准维护”。导入后的系列、变体、规格、资料、报价批次、报价行、阶梯价、选配项，最终以系列详情页和报价批次详情页作为长期维护入口。
- Priority 4 负责“统一数据口径和前台展示规则”。Priority 2 和 Priority 3 产生或维护的数据，都必须遵守 Priority 4 定义的状态口径、查询聚合口径和动态生成规则。

开发边界：

- Priority 2 可以在导入时创建商品、报价、阶梯价、资料和选配项草稿，但不应在工作台里实现完整的长期详情维护页面。
- Priority 2 的规格结构化只负责“从 Excel 生成规格项”；规格项的长期编辑、排序、分组和前台预览，归 Priority 3 的系列详情页处理。
- Priority 2 的报价选配项结构化只负责“从 Excel 生成选配项”；选配项的长期维护，归 Priority 3 的报价批次详情页处理。
- Priority 2 的多文件资料补齐只负责“入库流程中的快速补齐”；资料的长期维护，归 Priority 3 的系列详情页或现有文件资料页处理。
- Priority 2 不单独定义前台可见规则、动态生成规则、active / draft / archived / expired 的展示口径，这些统一由 Priority 4 治理。
- Priority 2 如需触发新品发布、报价更新、资料更新等前台动态，只能提供动作标记或预览入口，最终是否生成动态、生成哪类动态，必须遵守 Priority 4 的动态治理规则。

推荐执行关系：

1. 先增强 Priority 2，把 Excel 快速入库、重复型号处理、报价可编辑、附件留档、预检报告等入口能力做稳。
2. 再推进 Priority 3，把导入后的数据沉淀到系列详情页和报价批次详情页，作为长期维护中心。
3. 最后通过 Priority 4 统一查询层、状态口径和动态规则，避免前后台显示不一致。

### 2.1 导入预检报告

目标：在真正写入数据库前，给管理员一个清晰的导入风险与结果预览，避免误导入、重复导入和部分数据缺失。

待开发：

- `[ ]` 新增“导入预检报告”卡片。
- `[ ]` 统计本次 Excel 将新增多少个商品。
- `[ ]` 统计本次 Excel 将更新多少个已有商品。
- `[ ]` 统计本次 Excel 将只新增多少条报价批次。
- `[ ]` 统计本次 Excel 将创建多少条报价阶梯价。
- `[ ]` 统计缺分类、缺品牌、缺公司、缺报价、缺规格、缺图片的数量。
- `[ ]` 统计重复型号数量，并按处理策略分组展示。
- `[ ]` 支持“只导入正常行”。
- `[ ]` 支持“查看异常行”。
- `[ ]` 支持“下载错误报告”。
- `[ ]` 入库前弹出二次确认弹窗，展示导入计划摘要。

预检报告示例：

```text
商品：新增 6 个，更新 2 个，跳过 1 个
报价：新增批次 1 个，新增报价行 6 条，新增阶梯价 24 条
资料：保存原始 Excel 1 份，待提取图片 6 张
风险：重复型号 2 个，缺公司 1 个，缺报价 1 个
```

验收标准：

- 管理员在点击“确认入库”前，可以明确知道本次会新增、更新、跳过哪些数据。
- 异常行不会阻塞正常行入库。
- 预检结果与实际入库结果数量一致。

## Priority 3：改成“系列中心 + 批次中心”的工作方式

目标：让后台结构和前台展示结构一致，避免继续围绕单变体和平铺表格扩展。

- `[ ]` 规划 `/admin/products/:seriesId` 系列详情页，统一维护系列基础信息、变体、规格、图片、资料和关联公司。
- `[ ]` 规划 `/admin/quotes/:batchId` 报价批次详情页，统一维护批次头信息、报价行、阶梯价、选配项和附件。
- `[ ]` 前台产品详情改为系列详情，不再以单变体页面作为长期主形态。
- `[ ]` 明确“系列”和“变体”的前后台展示规则：前台详情主对象是系列，变体作为切换器；后台录入主对象也是系列。
- `[ ]` 补齐 `quote_options` 的后台维护入口，不能只读不写。

建议实施方案：

1. 后台产品列表页保留“系列列表”。
2. 点进系列后，在一个页面内完成：系列信息维护、变体列表维护、规格维护、图片和资料维护、前台预览。
3. 后台报价列表页保留“批次列表”。
4. 点进批次后，在一个页面内完成：批次信息、多变体报价行、阶梯价、选配项、附件、动态预览。

验收标准：

- 后台产品和报价的主对象与数据模型保持一致。
- 前台详情和后台详情围绕同一业务对象，不再各自维护一套理解。

## Priority 4：统一查询层和动态治理

目标：减少前后台两套口径继续漂移，并让动态只展示对前台真正有意义的信息。

### 4.1 查询层统一

- `[ ]` 梳理 `apps/web-front/src/api/product-info.ts` 与 `apps/web-ele/src/api/product-info/*` 的重复逻辑。
- `[ ]` 设计统一的聚合查询层或共享查询模块，统一产品、详情、报价、资料、动态的口径。
- `[ ]` 统一“active / archived / expired / draft”的前后台解释，避免一个页面显示、另一个页面隐藏。

建议方案：

1. 后台 API 负责写操作和管理态列表。
2. 前台 API 负责公开态聚合查询，但底层字段映射和状态规则抽到共享层。
3. 系列详情、报价批次详情、动态卡片都走统一聚合模型。

### 4.2 动态治理

- `[ ]` 重写动态生成规则，只保留“新品发布 / 有效报价更新 / 可见资料更新 / 管理员通知”四类。
- `[ ]` 取消“仅因报价附件上传就生成报价动态”的旧逻辑。
- `[ ]` 把“发布报价批次”作为报价动态的主要来源。
- `[ ]` 增加动态可见性规则：停用系列、停用变体、不可见资料、无效报价不进入前台动态流。
- `[ ]` 后台增加“是否生成前台动态”的确认或预览能力。

验收标准：

- 前台动态不再出现重复、噪音、误导信息。
- 动态内容能真实反映“产品发布、资料上新、报价更新、平台通知”。
